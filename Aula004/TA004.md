## 1. Conceitos Fundamentais

### O que são Servidores Web?

Um **servidor web** é um software que recebe requisições HTTP de clientes (navegadores) e responde com páginas web, arquivos ou dados. É como um garçom em um restaurante - ele recebe pedidos dos clientes e traz o que foi solicitado da cozinha (aplicação).

#### Principais Servidores Web:
- **Apache HTTP Server**: O mais antigo e popular
- **Nginx**: Focado em performance e proxy reverso
- **IIS**: Servidor da Microsoft para Windows
- **LiteSpeed**: Alternativa comercial de alta performance

### Nginx: O Servidor Web Moderno

#### Por que Nginx?
O **Nginx** (pronuncia-se "engine-x") foi criado para resolver o problema C10K - servir 10.000 conexões simultâneas. Ele é:
- **Rápido**: Arquitetura assíncrona e orientada a eventos
- **Leve**: Usa pouca memória RAM
- **Versátil**: Servidor web, proxy reverso, load balancer
- **Estável**: Usado por sites como Netflix, Airbnb, WordPress.com

#### Arquitetura do Nginx
Diferente do Apache que cria um processo/thread por conexão, o Nginx usa:
- **Master Process**: Gerencia configurações e workers
- **Worker Processes**: Processam requisições de forma assíncrona
- **Event Loop**: Um worker pode atender milhares de conexões

### Proxy Reverso: O Intermediário Inteligente

#### O que é um Proxy Reverso?
Um **proxy reverso** fica entre os clientes e os servidores de aplicação, funcionando como um intermediário que:
- Recebe requisições dos clientes
- Encaminha para servidores backend
- Retorna as respostas aos clientes

#### Analogia Prática: Recepcionista de Hotel
Imagine um hotel grande:
- **Cliente**: Quer falar com alguém do hotel
- **Recepcionista (Proxy Reverso)**: Recebe o pedido e direciona para o departamento correto
- **Departamentos (Servidores Backend)**: Processam o pedido específico

#### Proxy vs Proxy Reverso:

**Proxy Tradicional (Forward Proxy):**
```
Cliente → Proxy → Internet → Servidor
```
- Cliente sabe que está usando proxy
- Usado para controlar acesso à internet
- Exemplo: Proxy corporativo

**Proxy Reverso:**
```
Cliente → Internet → Proxy Reverso → Servidor Backend
```
- Cliente não sabe que existe um proxy
- Usado para distribuir carga e melhorar performance
- Exemplo: Nginx na frente de aplicações Node.js

### Configuração do Nginx

#### Estrutura de Configuração
O Nginx usa arquivos de configuração em formato específico:

```nginx
# Configuração global
user nginx;
worker_processes auto;

# Bloco de eventos
events {
    worker_connections 1024;
}

# Bloco HTTP
http {
    # Configurações HTTP globais
    
    server {
        # Configurações de um servidor virtual
        listen 80;
        server_name exemplo.com;
        
        location / {
            # Configurações de localização
        }
    }
}
```

#### Conceitos Principais:

##### 1. Server Blocks (Virtual Hosts)
Permitem hospedar múltiplos sites no mesmo servidor:

```nginx
server {
    listen 80;
    server_name site1.com;
    root /var/www/site1;
}

server {
    listen 80;
    server_name site2.com;
    root /var/www/site2;
}
```

##### 2. Location Blocks
Definem como processar diferentes URLs:

```nginx
location / {
    # Servir arquivos estáticos
    try_files $uri $uri/ =404;
}

location /api/ {
    # Proxy para aplicação backend
    proxy_pass http://localhost:3000;
}

location ~* \.(jpg|jpeg|png|gif)$ {
    # Cache para imagens
    expires 1y;
}
```

##### 3. Upstream (Grupos de Servidores)
Definem grupos de servidores backend:

```nginx
upstream backend {
    server 127.0.0.1:3000;
    server 127.0.0.1:3001;
    server 127.0.0.1:3002;
}

server {
    location / {
        proxy_pass http://backend;
    }
}
```

### Variáveis de Ambiente: Configuração Flexível

#### O que são Variáveis de Ambiente?
**Variáveis de ambiente** são valores que podem ser definidos fora do código da aplicação, permitindo diferentes configurações sem alterar o código fonte.

#### Por que usar Variáveis de Ambiente?
- **Segurança**: Senhas não ficam no código
- **Flexibilidade**: Mesma aplicação, configurações diferentes
- **Portabilidade**: Funciona em dev, teste e produção
- **Facilidade**: Mudanças sem recompilar código

#### Exemplo Prático:
```javascript
// ❌ Ruim: valores fixos no código
const dbHost = 'localhost';
const dbPassword = 'senha123';

// ✅ Bom: usando variáveis de ambiente
const dbHost = process.env.DB_HOST || 'localhost';
const dbPassword = process.env.DB_PASSWORD;
```

### Arquivo .env: Organizando Configurações

#### Estrutura do Arquivo .env:
```env
# Banco de dados
DB_HOST=localhost
DB_PORT=3306
DB_NAME=minha_aplicacao
DB_USER=usuario
DB_PASSWORD=senha_super_secreta

# Aplicação
APP_PORT=3000
APP_ENV=development
APP_DEBUG=true

# APIs externas
API_KEY=abc123xyz789
JWT_SECRET=meu_jwt_secreto
```

#### Boas Práticas para .env:
1. **Nunca commitar no Git**: Adicione `.env` no `.gitignore`
2. **Criar .env.example**: Template sem valores sensíveis
3. **Documentar variáveis**: Comentários explicativos
4. **Validar variáveis**: Verificar se existem ao iniciar app

### Load Balancing: Distribuindo a Carga

#### O que é Load Balancing?
**Load Balancing** é a distribuição de requisições entre múltiplos servidores para:
- Evitar sobrecarga de um servidor
- Melhorar performance geral
- Aumentar disponibilidade
- Permitir manutenção sem downtime

#### Métodos de Load Balancing:

##### 1. Round Robin (Padrão)
Distribui requisições sequencialmente:
```nginx
upstream backend {
    server app1:3000;
    server app2:3000;
    server app3:3000;
}
```
Requisições: app1 → app2 → app3 → app1 → ...

##### 2. Least Connections
Direciona para servidor com menos conexões ativas:
```nginx
upstream backend {
    least_conn;
    server app1:3000;
    server app2:3000;
}
```

##### 3. IP Hash
Mesmo cliente sempre vai para o mesmo servidor:
```nginx
upstream backend {
    ip_hash;
    server app1:3000;
    server app2:3000;
}
```

##### 4. Weighted (Ponderado)
Servidores com pesos diferentes:
```nginx
upstream backend {
    server app1:3000 weight=3;
    server app2:3000 weight=1;
}
```
app1 recebe 3x mais requisições que app2.

### SSL/TLS e HTTPS

#### Por que HTTPS é Importante?
- **Criptografia**: Dados não podem ser interceptados
- **Autenticidade**: Garante que você está no site correto
- **Integridade**: Dados não foram modificados
- **SEO**: Google favorece sites HTTPS
- **Confiança**: Usuários confiam mais

#### Configuração HTTPS no Nginx:
```nginx
server {
    listen 443 ssl;
    server_name exemplo.com;
    
    ssl_certificate /path/to/certificate.crt;
    ssl_certificate_key /path/to/private.key;
    
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
}

# Redirect HTTP para HTTPS
server {
    listen 80;
    server_name exemplo.com;
    return 301 https://$server_name$request_uri;
}
```

### Logs e Monitoramento

#### Tipos de Logs do Nginx:

##### 1. Access Log
Registra todas as requisições:
```nginx
log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                '$status $body_bytes_sent "$http_referer" '
                '"$http_user_agent"';

access_log /var/log/nginx/access.log main;
```

##### 2. Error Log
Registra erros e problemas:
```nginx
error_log /var/log/nginx/error.log warn;
```

#### Monitoramento com Nginx Status:
```nginx
location /nginx_status {
    stub_status on;
    access_log off;
    allow 127.0.0.1;
    deny all;
}
```

### Casos de Uso Práticos

#### 1. Aplicação Node.js com Nginx
```nginx
server {
    listen 80;
    server_name meuapp.com;
    
    # Arquivos estáticos
    location /static/ {
        root /var/www;
        expires 1y;
    }
    
    # API
    location / {
        proxy_pass http://localhost:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

#### 2. Microserviços com Roteamento
```nginx
server {
    listen 80;
    
    location /api/users/ {
        proxy_pass http://user-service:3001;
    }
    
    location /api/orders/ {
        proxy_pass http://order-service:3002;
    }
    
    location /api/products/ {
        proxy_pass http://product-service:3003;
    }
}
```

#### 3. SPA (Single Page Application)
```nginx
server {
    listen 80;
    root /var/www/spa;
    index index.html;
    
    # Tentar arquivo, depois pasta, senão index.html
    location / {
        try_files $uri $uri/ /index.html;
    }
    
    # API backend
    location /api/ {
        proxy_pass http://backend:3000;
    }
}
```

## 2. Questões de Múltipla Escolha

### Questão 1
Qual é a principal função de um proxy reverso?

a) Esconder a identidade do cliente na internet
b) Receber requisições de clientes e encaminhar para servidores backend
c) Bloquear acesso a sites maliciosos
d) Acelerar a conexão com a internet

**Gabarito: B**

**Justificativa:** Um proxy reverso atua como intermediário entre clientes e servidores backend, recebendo requisições dos clientes e encaminhando para os servidores apropriados. Isso permite load balancing, cache, SSL termination e outras funcionalidades. A opção A descreve um proxy tradicional (forward proxy), C descreve um firewall, e D não é uma função específica de proxy reverso.

### Questão 2
Por que é recomendado usar variáveis de ambiente em aplicações?

a) Para tornar o código mais rápido
b) Para reduzir o tamanho dos arquivos
c) Para separar configurações do código e aumentar a segurança
d) Para facilitar a compilação do programa

**Gabarito: C**

**Justificativa:** Variáveis de ambiente permitem separar configurações (como senhas, URLs de banco de dados) do código fonte, aumentando a segurança (senhas não ficam expostas no código) e flexibilidade (mesma aplicação pode ter configurações diferentes em dev/prod). As outras opções não são benefícios das variáveis de ambiente.

### Questão 3
No método de load balancing "Round Robin", como as requisições são distribuídas?

a) Para o servidor com menor uso de CPU
b) Sequencialmente entre os servidores disponíveis
c) Para o servidor geograficamente mais próximo
d) Baseado no endereço IP do cliente

**Gabarito: B**

**Justificativa:** No método Round Robin, as requisições são distribuídas sequencialmente entre os servidores disponíveis (servidor1 → servidor2 → servidor3 → servidor1...). É o método mais simples e padrão na maioria dos load balancers. A opção A descreve load balancing baseado em recursos, C descreve roteamento geográfico, e D descreve o método IP Hash.
