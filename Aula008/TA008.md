## 1. Conceitos Fundamentais

### O que é Amazon Web Services (AWS)?

**Amazon Web Services (AWS)** é a plataforma de computação em nuvem mais abrangente e amplamente adotada do mundo. Lançada em 2006, a AWS oferece mais de 200 serviços de data centers globalmente. É como ter acesso a um "supermercado tecnológico" onde você pode escolher exatamente os recursos que precisa, pagando apenas pelo que usar.

#### Analogia: AWS como uma Cidade Inteligente
Imagine a AWS como uma cidade moderna e inteligente:
- **Energia elétrica** = Computação (EC2)
- **Sistema de água** = Armazenamento (S3)
- **Correios** = Mensageria (SQS, SNS)
- **Bancos** = Banco de dados (RDS)
- **Sistema de segurança** = IAM e Security Groups
- **Transporte público** = Redes (VPC)

Você não precisa construir sua própria usina elétrica ou sistema de água - simplesmente "se conecta" aos serviços da cidade e paga pelo que consome.

### Infraestrutura Global da AWS

#### Regiões (Regions)
Uma **Região AWS** é uma área geográfica que contém múltiplos data centers isolados e fisicamente separados. Cada região é completamente independente das outras.

**Características das Regiões:**
- **Isolamento**: Falhas em uma região não afetam outras
- **Latência**: Escolha a região mais próxima dos seus usuários
- **Compliance**: Algumas regiões atendem requisitos específicos de conformidade
- **Serviços**: Nem todos os serviços estão disponíveis em todas as regiões

**Principais Regiões:**
- **us-east-1** (Norte da Virgínia): Região "padrão", mais serviços
- **us-west-2** (Oregon): Popular para aplicações na costa oeste
- **eu-west-1** (Irlanda): Principal região europeia
- **ap-southeast-1** (Singapura): Hub para Ásia-Pacífico
- **sa-east-1** (São Paulo): Única região no Brasil

#### Zonas de Disponibilidade (Availability Zones - AZs)
Cada região contém múltiplas **Zonas de Disponibilidade**, que são data centers fisicamente separados dentro da mesma região.

**Características das AZs:**
- **Isolamento físico**: Diferentes prédios, energia, refrigeração
- **Conectividade**: Interconectadas por redes de alta velocidade
- **Redundância**: Projetadas para tolerância a falhas
- **Latência baixa**: Comunicação rápida entre AZs da mesma região

**Exemplo prático:**
```
Região us-east-1 (Norte da Virgínia):
├── us-east-1a (AZ A)
├── us-east-1b (AZ B)
├── us-east-1c (AZ C)
├── us-east-1d (AZ D)
├── us-east-1e (AZ E)
└── us-east-1f (AZ F)
```

#### Edge Locations (Locais de Borda)
**Edge Locations** são pontos de presença da AWS espalhados pelo mundo, usados principalmente para:
- **CloudFront** (CDN - Content Delivery Network)
- **Route 53** (DNS)
- **AWS Global Accelerator**

São como "postos avançados" que aproximam o conteúdo dos usuários finais.

### Console AWS: Sua Central de Comando

#### O que é o Console AWS?
O **Console AWS** é a interface web que permite gerenciar todos os serviços AWS. É como o "painel de controle" da sua infraestrutura na nuvem.

#### Navegação no Console:
- **Barra de serviços**: Acesso rápido aos serviços mais usados
- **Região**: Seletor de região no canto superior direito
- **Conta**: Informações da conta e billing
- **Suporte**: Acesso ao suporte técnico

#### Principais Seções:
1. **Compute**: EC2, Lambda, ECS, EKS
2. **Storage**: S3, EBS, EFS
3. **Database**: RDS, DynamoDB, ElastiCache
4. **Networking**: VPC, CloudFront, Route 53
5. **Security**: IAM, KMS, Certificate Manager

### IAM: Identity and Access Management

#### O que é IAM?
**IAM** é o serviço que controla quem pode acessar o quê na sua conta AWS. É como o sistema de segurança de um prédio corporativo - define quem pode entrar, em quais andares, e o que pode fazer em cada local.

#### Componentes Principais do IAM:

##### 1. **Usuários (Users)**
Representam pessoas ou aplicações que precisam acessar a AWS:

```json
{
  "UserName": "joao.silva",
  "Path": "/developers/",
  "CreateDate": "2024-01-15T10:30:00Z",
  "UserId": "AIDACKCEVSQ6C2EXAMPLE"
}
```

##### 2. **Grupos (Groups)**
Coleções de usuários com permissões similares:

```json
{
  "GroupName": "Developers",
  "Path": "/",
  "GroupId": "AGPAI23HZ27SI6FQMGNQ2",
  "CreateDate": "2024-01-15T10:30:00Z"
}
```

##### 3. **Roles (Funções)**
Permissões que podem ser "assumidas" temporariamente:

```json
{
  "RoleName": "EC2-S3-Access",
  "Path": "/",
  "RoleId": "AROA12345678901234567",
  "AssumeRolePolicyDocument": "..."
}
```

##### 4. **Políticas (Policies)**
Documentos JSON que definem permissões:

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:PutObject"
      ],
      "Resource": "arn:aws:s3:::meu-bucket/*"
    }
  ]
}
```

#### Princípios de Segurança do IAM:

##### 1. **Princípio do Menor Privilégio**
Conceda apenas as permissões mínimas necessárias:

```json
// ❌ Muito permissivo
{
  "Effect": "Allow",
  "Action": "*",
  "Resource": "*"
}

// ✅ Específico e limitado
{
  "Effect": "Allow",
  "Action": [
    "s3:GetObject"
  ],
  "Resource": "arn:aws:s3:::meu-bucket-especifico/*"
}
```

##### 2. **Rotação Regular de Credenciais**
- Altere senhas regularmente
- Rotacione access keys
- Use credenciais temporárias quando possível

##### 3. **Multi-Factor Authentication (MFA)**
Adicione uma camada extra de segurança:
- Algo que você sabe (senha)
- Algo que você tem (token/celular)

##### 4. **Monitoramento e Auditoria**
- Use CloudTrail para logs de API
- Configure alertas para atividades suspeitas
- Revise permissões regularmente

### Modelos de Serviço na Nuvem

#### IaaS (Infrastructure as a Service)
**Infraestrutura como Serviço** - Você aluga a infraestrutura básica:

**Exemplos AWS:**
- **EC2**: Máquinas virtuais
- **VPC**: Redes virtuais
- **EBS**: Armazenamento em bloco

**Analogia**: Alugar um terreno e construir sua casa

**Responsabilidades:**
- **AWS**: Hardware, virtualização, data center
- **Cliente**: SO, aplicações, dados, configurações

#### PaaS (Platform as a Service)
**Plataforma como Serviço** - Você usa uma plataforma pronta para desenvolvimento:

**Exemplos AWS:**
- **Elastic Beanstalk**: Deploy de aplicações
- **RDS**: Banco de dados gerenciado
- **Lambda**: Computação serverless

**Analogia**: Alugar um apartamento mobiliado

**Responsabilidades:**
- **AWS**: Hardware, SO, runtime, middleware
- **Cliente**: Aplicações, dados, configurações

#### SaaS (Software as a Service)
**Software como Serviço** - Você usa software pronto:

**Exemplos AWS:**
- **WorkSpaces**: Desktop virtual
- **WorkMail**: Email corporativo
- **Chime**: Videoconferência

**Analogia**: Usar um hotel - tudo está pronto

**Responsabilidades:**
- **AWS**: Tudo, exceto dados do usuário
- **Cliente**: Apenas usar o serviço

### Modelos de Implantação

#### Nuvem Pública
- **Definição**: Recursos compartilhados entre múltiplos clientes
- **Vantagens**: Custo baixo, escalabilidade, sem manutenção
- **Desvantagens**: Menos controle, questões de compliance
- **Exemplo**: AWS padrão

#### Nuvem Privada
- **Definição**: Recursos dedicados a uma única organização
- **Vantagens**: Controle total, segurança, compliance
- **Desvantagens**: Custo alto, manutenção complexa
- **Exemplo**: AWS Outposts

#### Nuvem Híbrida
- **Definição**: Combinação de nuvem pública e privada
- **Vantagens**: Flexibilidade, otimização de custos
- **Desvantagens**: Complexidade de gerenciamento
- **Exemplo**: AWS Direct Connect + VPC

#### Multi-Cloud
- **Definição**: Uso de múltiplos provedores de nuvem
- **Vantagens**: Evita vendor lock-in, otimização por serviço
- **Desvantagens**: Complexidade, custos de integração
- **Exemplo**: AWS + Azure + Google Cloud

### Precificação da AWS

#### Modelos de Cobrança:

##### 1. **Pay-as-you-go (Pague pelo que usar)**
- Sem compromissos de longo prazo
- Sem custos iniciais
- Pague apenas pelos recursos consumidos

##### 2. **Reserved Instances (Instâncias Reservadas)**
- Compromisso de 1 ou 3 anos
- Desconto de até 75%
- Pagamento antecipado opcional

##### 3. **Spot Instances**
- Instâncias com desconto de até 90%
- Podem ser interrompidas pela AWS
- Ideais para workloads flexíveis

##### 4. **Dedicated Hosts**
- Servidores físicos dedicados
- Compliance e licenciamento
- Custo mais alto

#### Fatores que Afetam o Custo:
- **Região**: Preços variam por região
- **Tipo de instância**: Diferentes configurações
- **Armazenamento**: Tipo e quantidade
- **Transferência de dados**: Entre regiões e para internet
- **Tempo de uso**: Horas de execução

#### Ferramentas de Controle de Custos:
- **AWS Cost Explorer**: Análise de gastos
- **AWS Budgets**: Alertas de orçamento
- **AWS Cost and Usage Report**: Relatórios detalhados
- **AWS Trusted Advisor**: Recomendações de otimização

### Free Tier: Começando Gratuitamente

#### Tipos de Free Tier:

##### 1. **12 Meses Gratuitos**
Válido por 12 meses após criar a conta:
- **EC2**: 750 horas/mês de t2.micro
- **S3**: 5 GB de armazenamento
- **RDS**: 750 horas/mês de db.t2.micro

##### 2. **Sempre Gratuito**
Sem limite de tempo:
- **Lambda**: 1 milhão de requests/mês
- **DynamoDB**: 25 GB de armazenamento
- **CloudWatch**: 10 métricas customizadas

##### 3. **Testes Gratuitos**
Por tempo limitado:
- **Redshift**: 2 meses gratuitos
- **ElastiCache**: 750 horas/mês por 12 meses

#### Cuidados com o Free Tier:
- **Monitore o uso**: Configure alertas de billing
- **Entenda os limites**: Exceder pode gerar cobrança
- **Limpe recursos**: Delete recursos não utilizados
- **Configure budgets**: Defina limites de gastos

## 2. Questões de Múltipla Escolha

### Questão 1
Qual é a principal diferença entre uma Região e uma Zona de Disponibilidade na AWS?

a) Regiões são para backup e AZs são para produção
b) Regiões são áreas geográficas com múltiplas AZs, que são data centers isolados
c) AZs são maiores que Regiões e contêm múltiplos países
d) Não há diferença, são termos sinônimos

**Gabarito: B**

**Justificativa:** Regiões AWS são áreas geográficas (como "Norte da Virgínia" ou "São Paulo") que contêm múltiplas Zonas de Disponibilidade (AZs). Cada AZ é um data center fisicamente isolado dentro da região, projetado para tolerância a falhas. As outras opções estão incorretas: não há distinção de backup/produção (A), AZs são menores que regiões (C), e os termos não são sinônimos (D).

### Questão 2
O que é o princípio do "menor privilégio" no contexto do AWS IAM?

a) Dar acesso total inicialmente e remover permissões conforme necessário
b) Conceder apenas as permissões mínimas necessárias para realizar uma tarefa
c) Permitir acesso apenas aos serviços mais baratos da AWS
d) Restringir acesso apenas aos usuários com menor tempo de empresa

**Gabarito: B**

**Justificativa:** O princípio do menor privilégio é uma prática fundamental de segurança que recomenda conceder apenas as permissões mínimas necessárias para que um usuário ou sistema possa realizar suas tarefas. Isso reduz o risco de segurança e limita o impacto de possíveis comprometimentos. As outras opções não descrevem corretamente este princípio de segurança.

### Questão 3
Qual modelo de serviço em nuvem oferece maior controle sobre o sistema operacional e aplicações?

a) SaaS (Software as a Service)
b) PaaS (Platform as a Service)
c) IaaS (Infrastructure as a Service)
d) Todos oferecem o mesmo nível de controle

**Gabarito: C**

**Justificativa:** IaaS (Infrastructure as a Service) oferece o maior controle, pois fornece apenas a infraestrutura básica (máquinas virtuais, redes, armazenamento), deixando o cliente responsável pelo sistema operacional, middleware, runtime e aplicações. PaaS abstrai o SO e runtime, enquanto SaaS fornece software pronto para uso. Quanto mais "baixo" na pilha, maior o controle e responsabilidade do cliente.
