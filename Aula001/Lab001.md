## Passo 1: Preparação do Sistema
### Atualizando os Repositórios
Primeiro, vamos garantir que o sistema esteja atualizado:

```bash
sudo apt update
```

**O que faz:** Atualiza a lista de pacotes disponíveis nos repositórios configurados.

```bash
sudo apt upgrade -y
```

**O que faz:** Atualiza os pacotes já instalados para suas versões mais recentes.
- `-y`: Responde "yes" automaticamente para todas as perguntas

### Qual a diferença entre `apt update` e `apt upgrade`?

- **`apt update`**: Apenas baixa a lista atualizada de pacotes disponíveis. É como "verificar se há atualizações" - não instala nada.
- **`apt upgrade`**: Efetivamente instala as atualizações dos pacotes já instalados no sistema.

**Analogia:** É como verificar se há novos episódios de uma série (update) vs. efetivamente assistir aos episódios (upgrade).

## Passo 2: Instalação e Verificação

### Instalando o Nginx

```bash
sudo apt install nginx -y
```

**O que faz:** Instala o servidor web Nginx e suas dependências.
- `-y`: Confirma automaticamente a instalação

### Verificando o Status do Serviço

```bash
sudo systemctl status nginx
```

**O que você deve ver:** Uma saída similar a esta:
```
● nginx.service - A high performance web server and a reverse proxy server

   Active: active (running) since [data/hora]
```

**Pontos importantes:**
- **Active: active (running)** = Serviço funcionando corretamente
- **enabled** = Serviço iniciará automaticamente no boot

### Testando no Navegador

Abra seu navegador e acesse:
```

```

Você deve ver a página padrão "Welcome to nginx!"

### Entendendo o /var/www/html

O diretório `/var/www/html` é o **Document Root** padrão do Nginx no Ubuntu. Vamos entender sua estrutura:

```bash
ls -la /var/www/html
```

**O que você verá:**
```
total 12
drwxr-xr-x 2 root root 4096 [data] .
drwxr-xr-x 3 root root 4096 [data] ..
-rw-r--r-- 1 root root  612 [data] index.nginx-debian.html
```

### Decodificando as Permissões

- **drwxr-xr-x**: Permissões do diretório
- **root root**: Proprietário e grupo (ambos "root")
- **612**: Tamanho do arquivo em bytes

### Problema: Por que não usar sudo para tudo?

Usar `sudo` constantemente para editar arquivos web é:
- **Perigoso**: Risco de executar comandos destrutivos com privilégios elevados
- **Ineficiente**: Você não deveria precisar de permissões de administrador para desenvolver
- **Má prática**: Em produção, aplicações nunca rodam como root

### Solução: Transferindo a Propriedade

```bash
sudo chown -R $USER:$USER /var/www/html
```

**Explicação das flags:**
- `-R`: Recursivo (aplica a mudança a todos os arquivos e subdiretórios)
- `$USER`: Variável que contém seu nome de usuário atual
- `$USER:$USER`: Define usuário:grupo (ambos como seu usuário)

### Verificando a Mudança

```bash
ls -la /var/www/html
```

Agora você deve ver seu usuário como proprietário:
```
-rw-r--r-- 1 seuusuario seuusuario  612 [data] index.nginx-debian.html
```

## Passo 5: O Conceito de Virtual Host (Bônus Técnico)

### Como o Nginx Sabe Onde Buscar os Arquivos?

O Nginx usa arquivos de configuração chamados **Virtual Hosts** para determinar onde buscar os arquivos web.

```bash
cat /etc/nginx/sites-available/default
```

**Procure por esta linha importante:**
```nginx
root /var/www/html;
```

### Entendendo a Configuração

```nginx
server {
    listen 80 default_server;          # Escuta na porta 80 (HTTP)
    listen [::]:80 default_server;     # Escuta em IPv6
    
    root /var/www/html;                # Document Root - onde estão os arquivos
    index index.html index.htm;        # Arquivos padrão a serem servidos
    
    server_name _;                     # Nome do servidor (qualquer nome)
    
    location / {
        try_files $uri $uri/ =404;     # Como processar as requisições
    }
}
```

### Conceito Importante

- **Document Root**: O diretório base onde o servidor web procura pelos arquivos
- **Index Files**: Arquivos que são servidos automaticamente quando você acessa um diretório
- **Server Block**: Equivalente ao Virtual Host do Apache - define como um site é servido

> **Para o futuro:** Na AWS, você configurará algo similar, mas usando serviços como ALB (Application Load Balancer) e S3.

## Passo 6: Troubleshooting

### Monitorando Logs em Tempo Real

Quando algo der errado (e vai dar!), os logs são seus melhores amigos:

```bash
sudo tail -f /var/log/nginx/error.log
```

**Explicação das flags:**
- `-f`: Follow (acompanha o arquivo em tempo real)
- O comando ficará "travado" mostrando novos logs conforme aparecem

### Testando o Troubleshooting

1. **Deixe o comando `tail -f` rodando em um terminal**
2. **Abra outro terminal e execute:**

```bash
echo "HTML inválido <html><body><h1>Teste" > /var/www/html/teste.html
```

3. **No navegador, acesse:** `http://localhost/teste.html`
4. **Observe os logs no primeiro terminal**

### Logs de Acesso

Para ver quem está acessando seu site:

```bash
sudo tail -f /var/log/nginx/access.log
```

**Formato típico:**
```
127.0.0.1 - - [data/hora] "GET / HTTP/1.1" 200 1234 "-" "Mozilla/5.0..."
```

**Decodificando:**
- `127.0.0.1`: IP do cliente
- `GET /`: Método HTTP e caminho solicitado
- `200`: Código de status HTTP (200 = sucesso)
- `1234`: Tamanho da resposta em bytes

### Comandos Úteis para Troubleshooting

```bash
# Verificar se o Nginx está rodando
sudo systemctl status nginx

# Reiniciar o Nginx
sudo systemctl restart nginx

# Testar a configuração antes de reiniciar
sudo nginx -t

# Ver processos do Nginx
ps aux | grep nginx
```

## Glossário Técnico

- **Document Root**: Diretório base onde o servidor web procura pelos arquivos
- **Virtual Host**: Configuração que permite servir múltiplos sites em um servidor
- **systemctl**: Comando para gerenciar serviços no systemd (Ubuntu/Debian)
- **chown**: Comando para alterar proprietário de arquivos/diretórios
- **tail -f**: Comando para acompanhar arquivos de log em tempo real

## Problemas Comuns

### "Permission denied" ao editar arquivos
**Solução:** Verifique se executou o comando `chown` corretamente.

### Página não carrega
**Solução:** Verifique se o Nginx está rodando com `systemctl status nginx`.

### "Address already in use"
**Solução:** Outro serviço está usando a porta 80. Execute `sudo lsof -i :80` para identificar.

## 🧹 Seção de Limpeza

### Como Desfazer Tudo que Foi Feito

Se você quiser remover completamente tudo que foi instalado e configurado neste lab, siga os passos abaixo:

#### 1. Parando o Serviço Nginx

```bash
sudo systemctl stop nginx
sudo systemctl disable nginx
```

**O que faz:**
- `stop`: Para o serviço imediatamente
- `disable`: Impede que o serviço inicie automaticamente no boot

#### 2. Removendo o Nginx Completamente

```bash
# Remove o Nginx e suas dependências não utilizadas
sudo apt remove --purge nginx nginx-common -y
sudo apt autoremove -y
sudo apt autoclean
```

**Explicação das flags:**
- `--purge`: Remove o pacote e todos os arquivos de configuração
- `autoremove`: Remove dependências que não são mais necessárias
- `autoclean`: Limpa o cache de pacotes baixados

#### 3. Removendo Arquivos e Diretórios

```bash
# Remove os arquivos web criados
sudo rm -rf /var/www/html/*

# Remove a estrutura de pastas criada no home
rm -rf ~/ImplantacaoSistemas

# Remove logs do Nginx (opcional)
sudo rm -rf /var/log/nginx

# Remove diretórios de configuração restantes
sudo rm -rf /etc/nginx
```

**⚠️ ATENÇÃO:** O comando `rm -rf` é irreversível. Certifique-se de que está removendo apenas o que deseja.

#### 4. Verificando a Limpeza

```bash
# Verifica se o Nginx foi removido
which nginx
# Deve retornar: nginx not found

# Verifica se não há processos do Nginx rodando
ps aux | grep nginx
# Deve mostrar apenas o próprio comando grep

# Testa se a porta 80 está livre
sudo lsof -i :80
# Não deve mostrar nenhum processo
```

#### 5. Testando no Navegador

Acesse `http://localhost` - você deve ver uma mensagem de erro como:
- "This site can't be reached"
- "Connection refused"
- "Unable to connect"

Isso confirma que o servidor web foi removido com sucesso.

#### 6. Limpeza Final do Sistema

```bash
# Atualiza a lista de pacotes após remoções
sudo apt update

# Remove pacotes órfãos que podem ter sobrado
sudo apt autoremove --purge -y

# Limpa completamente o cache do apt
sudo apt clean
```

### Verificação Completa da Limpeza

Para ter certeza de que tudo foi removido:

```bash
# Verifica se não há arquivos relacionados ao Nginx
sudo find / -name "*nginx*" 2>/dev/null | head -10

# Verifica se não há usuários/grupos do Nginx
getent passwd | grep nginx
getent group | grep nginx
```

Se os comandos acima não retornarem nada, a limpeza foi bem-sucedida!

### Por que Fazer a Limpeza?

- **Economia de espaço**: Remove arquivos desnecessários
- **Segurança**: Elimina serviços que não estão sendo usados
- **Organização**: Mantém o sistema limpo para próximos labs
- **Aprendizado**: Entender como reverter configurações é importante

> **Dica:** Em ambientes de produção, sempre documente os procedimentos de instalação E remoção de serviços.
