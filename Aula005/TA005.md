## 1. Conceitos Fundamentais

### O que é Automação de Build?

**Automação de Build** é o processo de transformar código fonte em uma aplicação executável de forma automatizada, sem intervenção manual. É como ter uma fábrica robotizada que pega os ingredientes (código) e produz o produto final (aplicação) seguindo sempre os mesmos passos.

#### Por que Automatizar?
Imagine que toda vez que você quisesse fazer um bolo, tivesse que:
1. Lembrar de todos os ingredientes
2. Medir cada quantidade manualmente
3. Seguir 50 passos na ordem exata
4. Torcer para não esquecer nada

Com automação, você aperta um botão e a "máquina de fazer bolo" faz tudo automaticamente, sempre igual, sempre certo!

### Build Process: Da Ideia ao Produto

#### Etapas Típicas de um Build:

##### 1. **Preparação do Ambiente**
- Instalar dependências
- Configurar variáveis de ambiente
- Limpar builds anteriores

##### 2. **Compilação/Transpilação**
- Transformar código fonte em executável
- TypeScript → JavaScript
- SASS → CSS
- Otimizar código

##### 3. **Testes Automatizados**
- Executar testes unitários
- Testes de integração
- Verificar cobertura de código

##### 4. **Empacotamento**
- Criar arquivos de distribuição
- Gerar imagens Docker
- Comprimir assets

##### 5. **Validação**
- Verificar integridade
- Executar healthchecks
- Validar configurações

### Scripts de Automação: Seus Assistentes Digitais

#### O que são Scripts de Build?
**Scripts de build** são arquivos que contêm uma sequência de comandos para automatizar o processo de construção da aplicação. São como receitas detalhadas que o computador pode seguir.

#### Tipos de Scripts:

##### 1. Shell Scripts (.sh)
Para sistemas Unix/Linux:
```bash
#!/bin/bash
echo "Iniciando build..."
npm install
npm run test
npm run build
docker build -t minha-app .
echo "Build concluído!"
```

##### 2. Batch Scripts (.bat)
Para Windows:
```batch
@echo off
echo Iniciando build...
npm install
npm run test
npm run build
docker build -t minha-app .
echo Build concluído!
```

##### 3. Scripts NPM (package.json)
Para projetos Node.js:
```json
{
  "scripts": {
    "build": "webpack --mode production",
    "test": "jest",
    "start": "node server.js",
    "dev": "nodemon server.js",
    "docker:build": "docker build -t minha-app .",
    "docker:run": "docker run -p 3000:3000 minha-app"
  }
}
```

##### 4. Makefile
Sistema universal de build:
```makefile
build:
	npm install
	npm run test
	npm run build

docker:
	docker build -t minha-app .
	docker run -p 3000:3000 minha-app

clean:
	rm -rf node_modules
	rm -rf dist
```

### Healthchecks: O Médico da Sua Aplicação

#### O que são Healthchecks?
**Healthchecks** são verificações automáticas que monitoram se sua aplicação está funcionando corretamente. É como ter um médico que verifica constantemente os sinais vitais do paciente.

#### Tipos de Healthchecks:

##### 1. **Liveness Probe**
Verifica se a aplicação está "viva":
```javascript
// Endpoint simples de saúde
app.get('/health', (req, res) => {
  res.status(200).json({ 
    status: 'OK', 
    timestamp: new Date().toISOString() 
  });
});
```

##### 2. **Readiness Probe**
Verifica se a aplicação está pronta para receber tráfego:
```javascript
app.get('/ready', async (req, res) => {
  try {
    // Verificar conexão com banco
    await database.ping();
    // Verificar serviços externos
    await externalAPI.ping();
    
    res.status(200).json({ status: 'Ready' });
  } catch (error) {
    res.status(503).json({ status: 'Not Ready', error: error.message });
  }
});
```

##### 3. **Startup Probe**
Verifica se a aplicação terminou de inicializar:
```javascript
let isStarted = false;

// Simular inicialização
setTimeout(() => {
  isStarted = true;
}, 10000); // 10 segundos para inicializar

app.get('/startup', (req, res) => {
  if (isStarted) {
    res.status(200).json({ status: 'Started' });
  } else {
    res.status(503).json({ status: 'Starting' });
  }
});
```

#### Healthchecks no Docker:
```dockerfile
FROM node:16-alpine

COPY . /app
WORKDIR /app
RUN npm install

# Healthcheck integrado
HEALTHCHECK --interval=30s --timeout=3s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:3000/health || exit 1

CMD ["npm", "start"]
```

#### Healthchecks no Docker Compose:
```yaml
services:
  app:
    build: .
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
  
  db:
    image: mysql:8.0
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      timeout: 20s
      retries: 10
```

### Monitoramento e Observabilidade

#### Logs Estruturados
Em vez de logs simples, use logs estruturados:

```javascript
// ❌ Log simples
console.log('Usuário logou');

// ✅ Log estruturado
logger.info('User login', {
  userId: 123,
  email: 'user@example.com',
  timestamp: new Date().toISOString(),
  ip: req.ip,
  userAgent: req.get('User-Agent')
});
```

#### Métricas de Aplicação
Colete métricas importantes:

```javascript
const metrics = {
  requests: 0,
  errors: 0,
  responseTime: [],
  activeUsers: 0
};

app.use((req, res, next) => {
  const start = Date.now();
  metrics.requests++;
  
  res.on('finish', () => {
    const duration = Date.now() - start;
    metrics.responseTime.push(duration);
    
    if (res.statusCode >= 400) {
      metrics.errors++;
    }
  });
  
  next();
});

// Endpoint de métricas
app.get('/metrics', (req, res) => {
  const avgResponseTime = metrics.responseTime.reduce((a, b) => a + b, 0) / metrics.responseTime.length;
  
  res.json({
    totalRequests: metrics.requests,
    totalErrors: metrics.errors,
    errorRate: (metrics.errors / metrics.requests) * 100,
    averageResponseTime: avgResponseTime,
    activeUsers: metrics.activeUsers
  });
});
```

### Ferramentas de Build Modernas

#### 1. **Webpack**
Bundler para aplicações JavaScript:
```javascript
// webpack.config.js
module.exports = {
  entry: './src/index.js',
  output: {
    path: path.resolve(__dirname, 'dist'),
    filename: 'bundle.js'
  },
  module: {
    rules: [
      {
        test: /\.js$/,
        use: 'babel-loader'
      },
      {
        test: /\.css$/,
        use: ['style-loader', 'css-loader']
      }
    ]
  }
};
```

#### 2. **Vite**
Build tool moderna e rápida:
```javascript
// vite.config.js
export default {
  build: {
    outDir: 'dist',
    minify: 'terser',
    rollupOptions: {
      output: {
        manualChunks: {
          vendor: ['react', 'react-dom']
        }
      }
    }
  }
};
```

#### 3. **Docker Multi-stage Build**
Otimizar imagens Docker:
```dockerfile
# Estágio de build
FROM node:16-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

# Estágio de produção
FROM node:16-alpine AS production
WORKDIR /app
COPY --from=builder /app/node_modules ./node_modules
COPY . .
RUN npm run build

# Estágio final
FROM nginx:alpine
COPY --from=production /app/dist /usr/share/nginx/html
```

### Pipeline de Build Completo

#### Exemplo de Pipeline Automatizado:

```bash
#!/bin/bash
# build.sh - Script completo de build

set -e  # Parar se algum comando falhar

echo "🚀 Iniciando pipeline de build..."

# 1. Limpeza
echo "🧹 Limpando ambiente..."
rm -rf dist/
rm -rf node_modules/

# 2. Instalação de dependências
echo "📦 Instalando dependências..."
npm ci

# 3. Linting (verificação de código)
echo "🔍 Verificando qualidade do código..."
npm run lint

# 4. Testes
echo "🧪 Executando testes..."
npm run test:coverage

# 5. Build da aplicação
echo "🏗️ Construindo aplicação..."
npm run build

# 6. Build da imagem Docker
echo "🐳 Construindo imagem Docker..."
docker build -t minha-app:latest .

# 7. Testes de integração
echo "🔗 Executando testes de integração..."
docker-compose -f docker-compose.test.yml up --abort-on-container-exit

# 8. Verificação de segurança
echo "🔒 Verificando vulnerabilidades..."
npm audit --audit-level moderate

# 9. Healthcheck da imagem
echo "❤️ Verificando saúde da aplicação..."
docker run -d --name test-app -p 3000:3000 minha-app:latest
sleep 10
curl -f http://localhost:3000/health || exit 1
docker stop test-app && docker rm test-app

echo "✅ Build concluído com sucesso!"
```

### Boas Práticas de Automação

#### 1. **Fail Fast**
Pare o build no primeiro erro:
```bash
set -e  # Bash
# ou
npm run build && npm run test && docker build .
```

#### 2. **Logs Detalhados**
Registre cada etapa:
```bash
log() {
  echo "[$(date +'%Y-%m-%d %H:%M:%S')] $1"
}

log "Iniciando build..."
log "Instalando dependências..."
```

#### 3. **Versionamento**
Use tags semânticas:
```bash
VERSION=$(git describe --tags --always)
docker build -t minha-app:$VERSION .
docker tag minha-app:$VERSION minha-app:latest
```

#### 4. **Cache Inteligente**
Aproveite cache quando possível:
```dockerfile
# Copiar package.json primeiro para aproveitar cache
COPY package*.json ./
RUN npm ci

# Depois copiar código fonte
COPY . .
RUN npm run build
```

#### 5. **Rollback Automático**
Prepare para reverter mudanças:
```bash
# Backup da versão anterior
docker tag minha-app:latest minha-app:backup

# Deploy nova versão
docker build -t minha-app:latest .

# Se healthcheck falhar, fazer rollback
if ! curl -f http://localhost:3000/health; then
  echo "Healthcheck falhou, fazendo rollback..."
  docker tag minha-app:backup minha-app:latest
  exit 1
fi
```

## 2. Questões de Múltipla Escolha

### Questão 1
Qual é o principal objetivo da automação de build?

a) Reduzir o tamanho dos arquivos da aplicação
b) Transformar código fonte em aplicação executável de forma consistente e automatizada
c) Melhorar a velocidade de execução da aplicação
d) Facilitar a instalação da aplicação no computador do usuário

**Gabarito: B**

**Justificativa:** A automação de build tem como objetivo principal transformar o código fonte em uma aplicação executável de forma consistente, repetível e sem intervenção manual, eliminando erros humanos e garantindo que o processo seja sempre o mesmo. As outras opções podem ser consequências da automação, mas não são o objetivo principal.

### Questão 2
O que é um healthcheck em uma aplicação?

a) Um teste que verifica se o código está bem escrito
b) Uma verificação automática que monitora se a aplicação está funcionando corretamente
c) Um backup automático dos dados da aplicação
d) Um sistema de atualização automática da aplicação

**Gabarito: B**

**Justificativa:** Healthcheck é uma verificação automática que monitora continuamente se a aplicação está funcionando corretamente, verificando aspectos como conectividade com banco de dados, disponibilidade de serviços externos, e responsividade da aplicação. Não é um teste de código (opção A), backup (opção C), ou sistema de atualização (opção D).

### Questão 3
Por que é recomendado usar "fail fast" em scripts de build?

a) Para acelerar o processo de build
b) Para economizar recursos do servidor
c) Para parar o build imediatamente quando ocorre um erro, evitando problemas maiores
d) Para reduzir o tamanho dos logs gerados

**Gabarito: C**

**Justificativa:** O princípio "fail fast" recomenda parar o processo de build imediatamente quando um erro é detectado, evitando que problemas se propaguem e causem falhas maiores ou mais difíceis de diagnosticar. Isso economiza tempo e facilita a identificação da causa raiz do problema. As outras opções não são os motivos principais para usar "fail fast".
