Ao final deste laboratório, você será capaz de:

## Passo 1: Instalação do Docker Engine

### Preparando o Sistema

Primeiro, vamos remover versões antigas e instalar dependências:

```bash
sudo apt update
sudo apt remove docker docker-engine docker.io containerd runc -y
sudo apt install ca-certificates curl gnupg lsb-release -y
```

### Adicionando o Repositório Oficial do Docker

```bash
sudo mkdir -m 0755 -p /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg
```

**O que faz:**
- `curl -fsSL`: Baixa o arquivo de forma silenciosa e segura
- `gpg --dearmor`: Converte a chave GPG para formato binário
- `/etc/apt/keyrings/`: Diretório padrão para chaves de repositórios

```bash
echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu $(lsb_release -cs) stable" | sudo tee /etc/apt/sources.list.d/docker.list > /dev/null
```

### Instalando o Docker

```bash
sudo apt update
sudo apt install docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin -y
```

### Comando Vital: Permitir Docker sem sudo

```bash
sudo usermod -aG docker $USER
```

**Por que isso é importante?**
- **Segurança**: Evita executar comandos Docker com privilégios de root desnecessariamente
- **Produtividade**: Não precisar digitar `sudo` a cada comando
- **Boas práticas**: Em ambientes de produção, aplicações nunca devem rodar como root
> [!IMPORTANT]
>**IMPORTANTE:** Após executar este comando, você precisa fazer logout e login novamente, ou executar:

```bash
newgrp docker
```

### Verificando a Instalação

```bash
docker --version
docker run hello-world
```

Se tudo estiver correto, você verá a mensagem "Hello from Docker!"

## Passo 2: O Primeiro Container

### Testando com Hello World

```bash
docker run hello-world
```

**O que aconteceu:**
1. Docker procurou a imagem `hello-world` localmente
2. Não encontrou, então baixou do Docker Hub
3. Criou um container baseado na imagem
4. Executou o container e mostrou a mensagem
5. Container finalizou automaticamente

### Rodando Nginx Oficial

```bash
docker run -d -p 8080:80 --name meu-nginx nginx
```

**Explicação exaustiva das flags:**
- `-d` (detach): Roda o container em background, liberando o terminal
- `-p 8080:80` (publish): Mapeia a porta 8080 do host para a porta 80 do container
  - `8080`: Porta no seu computador (host)
  - `80`: Porta dentro do container
- `--name meu-nginx`: Dá um nome amigável ao container (senão seria um ID aleatório)
- `nginx`: Nome da imagem a ser usada

### Testando o Container

Acesse `http://localhost:8080` - você deve ver a página padrão do Nginx!

### Verificando Containers Ativos

```bash
docker ps
```

**Saída esperada:**
```
CONTAINER ID   IMAGE   COMMAND                  CREATED         STATUS         PORTS                  NAMES
abc123def456   nginx   "/docker-entrypoint.…"   2 minutes ago   Up 2 minutes   0.0.0.0:8080->80/tcp   meu-nginx
```

### Criando o Dockerfile

```bash
code Dockerfile
```

**Digite o seguinte conteúdo:**

```dockerfile
# Usar a imagem base do Nginx Alpine
FROM nginx:alpine

# Copiar nosso HTML personalizado para o container
COPY index.html /usr/share/nginx/html/

# Expor a porta 80
EXPOSE 80

# O comando padrão já está definido na imagem base nginx:alpine
```

### Entendendo Cada Instrução

**FROM nginx:alpine**
- `FROM`: Define a imagem base para construir nossa imagem
- `nginx:alpine`: Versão do Nginx baseada no Alpine Linux
- **Por que Alpine?** É uma distribuição Linux minimalista (~5MB vs ~100MB+ de outras distros)
  - Menor tamanho = deploy mais rápido na Cloud
  - Menor superfície de ataque = mais seguro
  - Menos recursos consumidos = menor custo na AWS

**COPY index.html /usr/share/nginx/html/**
- Copia o arquivo `index.html` do diretório atual para dentro da imagem
- `/usr/share/nginx/html/`: Diretório padrão do Nginx no container

**EXPOSE 80**
- Documenta que o container escuta na porta 80
- **Importante**: Não publica a porta automaticamente, apenas documenta
- É uma boa prática para outros desenvolvedores entenderem quais portas usar

## Passo 4: Build e Tagging

### Construindo a Imagem

```bash
docker build -t meu-app-ads:v1 .
```

**Explicação detalhada:**
- `docker build`: Comando para construir uma imagem
- `-t meu-app-ads:v1`: Tag (etiqueta) da imagem
  - `meu-app-ads`: Nome da imagem
  - `v1`: Versão/tag específica
- `.` (ponto): **CRUCIAL** - indica o contexto de build (diretório atual)

### Por que o ponto (.) é importante?

O ponto define o **build context** - todos os arquivos que o Docker pode acessar durante o build:
- Docker envia todos os arquivos do contexto para o daemon
- Apenas arquivos no contexto podem ser copiados com `COPY` ou `ADD`
- Contextos grandes tornam o build lento

### Verificando a Imagem Criada

```bash

```

**Saída esperada:**

```
nginx         alpine    789abc123def   2 weeks ago      23.2MB
```

## Passo 5: Gerenciamento e Inspeção

### Listando Containers

```bash
# Containers ativos
docker ps

# Todos os containers (ativos e parados)
docker ps -a
```

### Parando Containers

```bash
docker stop meu-nginx
docker stop minha-app
```

### Diferença: Parar vs. Remover

**Parar um container:**
```bash
docker stop nome-do-container
```
- Container para de executar mas permanece no sistema
- Dados temporários são preservados
- Pode ser reiniciado com `docker start`

**Remover um container:**
```bash
docker rm nome-do-container
```
- Container é completamente removido do sistema
- Dados temporários são perdidos
- Não pode ser reiniciado (precisa criar novo)

### Visualizando Logs

```bash
# Logs de um container específico
docker logs minha-app

# Acompanhar logs em tempo real
docker logs -f minha-app
```

**Flags úteis:**
- `-f`: Follow (acompanha em tempo real)
- `--tail 50`: Mostra apenas as últimas 50 linhas
- `--since 1h`: Mostra logs da última hora

### Inspecionando Containers

```bash
# Informações detalhadas do container
docker inspect minha-app

# Apenas o IP do container
docker inspect -f '{{range .NetworkSettings.Networks}}{{.IPAddress}}{{end}}' minha-app
```

### Executando Comandos Dentro do Container

```bash
# Abrir um shell interativo
docker exec -it minha-app sh

# Executar um comando específico
docker exec minha-app ls -la /usr/share/nginx/html/
```

**Flags:**
- `-i`: Interativo (mantém STDIN aberto)
- `-t`: Aloca um pseudo-TTY (terminal)

## Passo 6: Desafio de Persistência (Volume)

### O Problema da Persistência

Containers são **efêmeros** - quando removidos, todos os dados são perdidos. Para desenvolvimento, queremos editar arquivos e ver mudanças imediatamente.

### Solução: Mapeamento de Volume

```bash
# Parar e remover container anterior
docker stop minha-app
docker rm minha-app

# Rodar com volume mapeado
docker run -d -p 8081:80 -v $(pwd):/usr/share/nginx/html --name minha-app-dev nginx:alpine
```

**Explicação da flag `-v`:**
- `-v $(pwd):/usr/share/nginx/html`: Mapeia diretório local para dentro do container
- `$(pwd)`: Diretório atual (onde está o index.html)
- `/usr/share/nginx/html`: Diretório dentro do container
- Mudanças no arquivo local aparecem instantaneamente no container

### Testando a Persistência

1. **Edite o index.html no VS Code ou nano**
2. **Altere o título para "Deploy com Docker - UniFAAT"**
3. **Salve o arquivo**
4. **Atualize o navegador em `http://localhost:8081`**

A mudança aparece imediatamente, sem precisar fazer novo build!

### Tipos de Volume

```bash
# Volume nomeado (gerenciado pelo Docker)
docker run -v meu-volume:/usr/share/nginx/html nginx:alpine

# Bind mount (mapeia diretório específico do host)
docker run -v /home/user/site:/usr/share/nginx/html nginx:alpine

# Volume temporário (tmpfs - apenas na memória)
docker run --tmpfs /tmp nginx:alpine
```


## Nota Técnica: Docker vs. Máquinas Virtuais

### Consumo de Recursos

| Aspecto | Máquina Virtual | Container Docker |
|---------|----------------|------------------|
| **RAM** | 512MB - 2GB+ por VM | 10MB - 100MB por container |
| **CPU** | Overhead do hypervisor | Compartilha kernel do host |
| **Disco** | 10GB+ por VM | 100MB - 1GB por imagem |
| **Boot Time** | 30s - 2min | 1s - 5s |
| **Isolamento** | Completo (kernel próprio) | Processo isolado |

### Vantagens dos Containers

- **Densidade**: Mais containers por servidor
- **Velocidade**: Start/stop quase instantâneo
- **Portabilidade**: "Funciona na minha máquina" = funciona em produção
- **Eficiência**: Compartilhamento de recursos otimizado
