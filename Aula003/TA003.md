## 1. Conceitos Fundamentais

### O que é Orquestração de Containers?

**Orquestração** é o processo de gerenciar múltiplos containers que trabalham juntos para formar uma aplicação completa. É como ser o maestro de uma orquestra - você coordena diferentes instrumentos (containers) para criar uma sinfonia harmoniosa (aplicação funcionando).

Imagine que você tem uma aplicação web moderna que precisa de:
- Um container para o frontend (React/Angular)
- Um container para o backend (Node.js/Python)
- Um container para o banco de dados (MySQL/PostgreSQL)
- Um container para cache (Redis)

Cada container é como um músico especializado, mas eles precisam tocar juntos no tempo certo!

### Docker Compose: O Maestro dos Containers

#### O que é Docker Compose?
**Docker Compose** é uma ferramenta que permite definir e executar aplicações Docker multi-container. Em vez de executar vários comandos `docker run` separados, você define tudo em um arquivo YAML e executa com um único comando.

#### Por que usar Docker Compose?
**Sem Docker Compose (modo difícil):**
```bash
# Criar rede
docker network create minha-rede

# Executar banco
docker run -d --name db --network minha-rede -e MYSQL_ROOT_PASSWORD=123 mysql

# Executar aplicação
docker run -d --name app --network minha-rede -p 3000:3000 minha-app

# Executar nginx
docker run -d --name nginx --network minha-rede -p 80:80 nginx
```

**Com Docker Compose (modo fácil):**
```bash
docker-compose up -d
```

### Arquivo docker-compose.yml: A Partitura da Orquestra

O arquivo `docker-compose.yml` é onde você define todos os serviços, redes e volumes da sua aplicação. É como uma partitura musical que diz a cada "músico" (container) o que fazer.

#### Estrutura Básica:
```yaml
version: '3.8'

services:
  app:
    # Configurações do container da aplicação
  
  db:
    # Configurações do container do banco
  
  nginx:
    # Configurações do container do proxy

networks:
  # Definição das redes

volumes:
  # Definição dos volumes
```

#### Componentes Principais:

##### 1. Services (Serviços)
Cada serviço representa um container. É como definir cada músico da orquestra:

```yaml
services:
  web:
    image: nginx:alpine
    ports:
      - "80:80"
  
  api:
    build: ./api
    ports:
      - "3000:3000"
    depends_on:
      - db
  
  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: senha123
```

##### 2. Networks (Redes)
Definem como os containers se comunicam entre si:

```yaml
networks:
  frontend:
    # Rede para comunicação web
  backend:
    # Rede para comunicação interna
```

##### 3. Volumes (Volumes)
Definem onde os dados são armazenados de forma persistente:

```yaml
volumes:
  db_data:
    # Volume para dados do banco
  app_logs:
    # Volume para logs da aplicação
```

### Persistência de Dados: Mantendo Informações Vivas

#### O Problema da Volatilidade
Por padrão, containers são **efêmeros** - quando você remove um container, todos os dados dentro dele são perdidos. É como escrever na areia da praia - quando a onda vem, tudo desaparece!

#### Tipos de Persistência no Docker:

##### 1. Volumes (Recomendado)
Volumes são gerenciados pelo Docker e ficam fora do container:

```yaml
services:
  db:
    image: mysql:8.0
    volumes:
      - db_data:/var/lib/mysql  # Volume nomeado

volumes:
  db_data:  # Definição do volume
```

**Vantagens dos Volumes:**
- Gerenciados pelo Docker
- Funcionam em qualquer sistema operacional
- Podem ser compartilhados entre containers
- Backup e restore mais fáceis

##### 2. Bind Mounts
Mapeiam uma pasta do host diretamente para o container:

```yaml
services:
  app:
    image: minha-app
    volumes:
      - ./codigo:/app  # Pasta local mapeada
      - ./logs:/var/log/app
```

**Quando usar Bind Mounts:**
- Durante desenvolvimento (código em tempo real)
- Arquivos de configuração
- Logs que você quer acessar facilmente

##### 3. tmpfs Mounts
Armazenam dados na memória RAM (temporário):

```yaml
services:
  cache:
    image: redis
    tmpfs:
      - /data  # Dados na RAM
```

### Redes Docker: Conectando os Containers

#### Tipos de Redes:

##### 1. Bridge (Padrão)
Containers na mesma rede bridge podem se comunicar:

```yaml
networks:
  default:
    driver: bridge
```

##### 2. Host
Container usa a rede do host diretamente:

```yaml
services:
  app:
    network_mode: host
```

##### 3. None
Container sem conectividade de rede:

```yaml
services:
  isolated:
    network_mode: none
```

#### Comunicação Entre Containers
Containers na mesma rede podem se comunicar usando o nome do serviço:

```yaml
services:
  app:
    # Pode acessar o banco usando: mysql://db:3306
  
  db:
    # Nome do serviço = hostname na rede
```

### Comandos Essenciais do Docker Compose

#### Comandos de Ciclo de Vida:
```bash
# Iniciar todos os serviços
docker-compose up

# Iniciar em background
docker-compose up -d

# Parar todos os serviços
docker-compose down

# Parar e remover volumes
docker-compose down -v

# Rebuild das imagens
docker-compose build

# Iniciar com rebuild
docker-compose up --build
```

#### Comandos de Monitoramento:
```bash
# Ver status dos serviços
docker-compose ps

# Ver logs de todos os serviços
docker-compose logs

# Ver logs de um serviço específico
docker-compose logs app

# Seguir logs em tempo real
docker-compose logs -f

# Executar comando em um serviço
docker-compose exec app bash
```

### Variáveis de Ambiente e Configuração

#### Arquivo .env
Você pode usar um arquivo `.env` para definir variáveis:

```env
# .env
MYSQL_ROOT_PASSWORD=senha_super_secreta
APP_PORT=3000
NODE_ENV=production
```

#### Usando no docker-compose.yml:
```yaml
services:
  db:
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
  
  app:
    ports:
      - "${APP_PORT}:3000"
    environment:
      NODE_ENV: ${NODE_ENV}
```

### Dependências Entre Serviços

#### depends_on: Ordem de Inicialização
```yaml
services:
  app:
    depends_on:
      - db
      - redis
  
  db:
    image: mysql:8.0
  
  redis:
    image: redis:alpine
```

#### Healthchecks: Dependência Inteligente
```yaml
services:
  db:
    image: mysql:8.0
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      timeout: 20s
      retries: 10
  
  app:
    depends_on:
      db:
        condition: service_healthy
```

### Casos de Uso Práticos

#### 1. Stack LAMP (Linux, Apache, MySQL, PHP)
```yaml
version: '3.8'
services:
  web:
    image: php:apache
    ports:
      - "80:80"
    volumes:
      - ./src:/var/www/html
    depends_on:
      - db
  
  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: senha123
    volumes:
      - db_data:/var/lib/mysql

volumes:
  db_data:
```

#### 2. Stack Node.js + MongoDB
```yaml
version: '3.8'
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      MONGO_URL: mongodb://db:27017/meuapp
    depends_on:
      - db
  
  db:
    image: mongo:latest
    volumes:
      - mongo_data:/data/db

volumes:
  mongo_data:
```

### Boas Práticas

#### 1. Organização de Arquivos
```
projeto/
├── docker-compose.yml
├── .env
├── app/
│   └── Dockerfile
├── nginx/
│   └── nginx.conf
└── data/
```

#### 2. Nomeação Consistente
- Use nomes descritivos para serviços
- Mantenha padrão de nomenclatura
- Documente configurações complexas

#### 3. Segurança
- Não coloque senhas no docker-compose.yml
- Use arquivo .env para variáveis sensíveis
- Configure redes isoladas quando necessário

#### 4. Performance
- Use volumes para dados que mudam frequentemente
- Configure limites de recursos quando necessário
- Use multi-stage builds para imagens menores

## 2. Questões de Múltipla Escolha

### Questão 1
Qual é a principal vantagem do Docker Compose em relação a executar containers individuais?

a) Docker Compose torna os containers mais seguros
b) Docker Compose permite gerenciar múltiplos containers como uma única aplicação
c) Docker Compose faz os containers rodarem mais rápido
d) Docker Compose reduz o uso de memória dos containers

**Gabarito: B**

**Justificativa:** A principal vantagem do Docker Compose é permitir definir e gerenciar múltiplos containers relacionados como uma única aplicação, facilitando a orquestração, configuração de redes, volumes e dependências entre serviços. Com um único comando (`docker-compose up`), você pode iniciar toda a stack da aplicação. As outras opções não são benefícios diretos do Compose.

### Questão 2
O que acontece com os dados armazenados dentro de um container quando ele é removido?

a) Os dados são automaticamente salvos no host
b) Os dados são perdidos permanentemente, a menos que estejam em volumes
c) Os dados são transferidos para outros containers
d) Os dados ficam disponíveis na imagem Docker

**Gabarito: B**

**Justificativa:** Containers são efêmeros por natureza - quando removidos, todos os dados armazenados em seu sistema de arquivos interno são perdidos permanentemente. Para persistir dados, é necessário usar volumes Docker ou bind mounts, que armazenam os dados fora do container. As outras opções são incorretas: dados não são salvos automaticamente, não são transferidos para outros containers, e não ficam na imagem.

### Questão 3
No Docker Compose, qual configuração permite que um serviço aguarde outro serviço estar funcionando antes de iniciar?

a) wait_for
b) depends_on
c) start_after
d) requires

**Gabarito: B**

**Justificativa:** A configuração `depends_on` no Docker Compose define dependências entre serviços, fazendo com que um serviço aguarde outros serviços iniciarem antes de começar sua própria inicialização. Pode ser usado de forma simples (aguarda apenas o start) ou com healthcheck (aguarda o serviço estar saudável). As outras opções não existem no Docker Compose.
