
### Diferença: docker-compose vs docker compose

| Aspecto | docker-compose (v1) | docker compose (v2) |
|---------|-------------------|-------------------|
| **Instalação** | Pacote Python separado | Plugin integrado ao Docker |
| **Comando** | `docker-compose up` | `docker compose up` |
| **Performance** | Mais lento | Mais rápido (escrito em Go) |
| **Manutenção** | Descontinuado | Versão atual e mantida |
| **Compatibilidade** | YAML v2/v3 | YAML v3+ com recursos extras |

> [!IMPORTANT]
> **Importante:** Sempre use `docker compose` (com espaço) - é a versão moderna e mantida.

## Passo 3: Escrevendo o YAML

### Criando o docker-compose.yml

```bash
code docker-compose.yml
```

**Digite o seguinte conteúdo:**

```yaml
version: '3.8'

services:
  web:
    image: nginx:alpine
    container_name: web-frontend
    ports:
      - "8080:80"
    volumes:
      - ./app:/usr/share/nginx/html
    networks:
      - app-network
    restart: always
    depends_on:
      - app

  app:
    image: httpd:alpine
    container_name: app-backend
    ports:
      - "8081:80"
    networks:
      - app-network
    restart: always
    environment:
      - ENV=development
      - SERVICE_NAME=backend-api

networks:
  app-network:
    driver: bridge

volumes:
  app-data:
    driver: local
```

### Explicação Detalhada de Cada Diretiva

**version: '3.8'**
- Define a versão do formato do Compose file
- Versão 3.8 suporta recursos modernos e é compatível com Docker Swarm

**services:**
- Define todos os containers que compõem sua aplicação
- Cada serviço é um container independente

**image: nginx:alpine**
- Especifica a imagem Docker a ser usada
- Alpine = distribuição Linux minimalista (~5MB)

**container_name: web-frontend**
- Nome personalizado para o container (opcional)
- Facilita identificação e debugging

**ports: - "8080:80"**
- Mapeia porta do host (8080) para porta do container (80)
- Formato: "HOST:CONTAINER"

**volumes: - ./app:/usr/share/nginx/html**
- Monta diretório local dentro do container
- Permite edição de arquivos sem rebuild

**networks: - app-network**
- Conecta o serviço à rede personalizada
- Permite comunicação entre containers por nome

**restart: always**
- Política de reinicialização automática
- Opções: no, always, on-failure, unless-stopped

**depends_on: - app**
- Define ordem de inicialização
- web só inicia depois que app estiver rodando

**environment:**
- Define variáveis de ambiente dentro do container
- Útil para configurações específicas por ambiente

**networks: app-network:**
- Cria rede personalizada para os serviços
- driver: bridge = rede local entre containers

### Cuidados com YAML

```yaml
# ✅ CORRETO - Identação com 2 espaços
services:
  web:
    image: nginx:alpine
    ports:
      - "8080:80"

# ❌ ERRADO - Mistura tabs e espaços
services:
	web:
    image: nginx:alpine
	  ports:
      - "8080:80"
```

> [!NOTE]
> **Dica do Professor:** YAML é sensível à identação! Use sempre 2 espaços, nunca tabs. Configure seu editor para mostrar espaços em branco.

## Passo 4: Ciclo de Vida do Compose

### Iniciando o Stack

```bash
docker compose up -d
```

**Flags importantes:**
- `-d`: Detached mode (roda em background)
- `--build`: Força rebuild das imagens
- `--force-recreate`: Recria containers mesmo se não houve mudanças

### Verificando os Serviços

```bash
docker compose ps
```

**Saída esperada:**
```
NAME            IMAGE          COMMAND                  SERVICE   CREATED         STATUS         PORTS
app-backend     httpd:alpine   "httpd-foreground"       app       2 minutes ago   Up 2 minutes   0.0.0.0:8081->80/tcp
web-frontend    nginx:alpine   "/docker-entrypoint.…"   web       2 minutes ago   Up 2 minutes   0.0.0.0:8080->80/tcp
```

### Visualizando Logs

```bash
# Logs de todos os serviços
docker compose logs -f

# Logs de um serviço específico
docker compose logs -f web

# Últimas 50 linhas
docker compose logs --tail=50
```

### Testando os Serviços

```bash
# Testar o frontend (Nginx)
curl http://localhost:8080

# Testar o backend (Apache)
curl http://localhost:8081
```

### Parando e Removendo o Stack

```bash
# Parar serviços (containers permanecem)
docker compose stop

# Iniciar serviços parados
docker compose start

# Parar e remover tudo
docker compose down
```

### Por que `down` é Mais Potente?

| Comando | Containers | Redes | Volumes |
|---------|------------|-------|---------|
| `docker compose stop` | Para | Mantém | Mantém |
| `docker compose down` | Remove | Remove | Mantém* |
| `docker compose down -v` | Remove | Remove | Remove |

*Volumes nomeados são preservados por padrão

## Passo 5: Rede e Comunicação Inter-container

### Entendendo a Rede Interna

Quando você cria serviços no Compose, eles podem se comunicar usando os nomes definidos no YAML como DNS.

### Testando Comunicação Entre Containers

```bash
# Entrar no container web
docker exec -it web-frontend sh
```

**Dentro do container, execute:**

```bash
# Testar conectividade com ping
ping app

# Fazer requisição HTTP para o serviço app
wget -qO- http://app

# Ou usando curl (se disponível)
curl http://app
```

### Verificando a Resolução DNS

```bash
# Ainda dentro do container web
nslookup app
```

**Saída esperada:**
```
Server:    127.0.0.11
Address:   127.0.0.11:53

Name:      app
Address:   172.18.0.3
```

### Inspecionando a Rede

```bash
# Sair do container
exit

# Listar redes Docker
docker network ls

# Inspecionar a rede do projeto
docker network inspect aula03_app-network
```

### Como Funciona o DNS Interno?

1. **Docker cria um DNS server interno** (127.0.0.11)
2. **Cada container recebe um IP** na rede personalizada
3. **Nomes dos serviços** se tornam hostnames válidos
4. **Resolução automática** entre containers na mesma rede

> [!NOTE]
> **Conceito Fundamental:** Esta é a base dos microsserviços! Serviços se comunicam por nome, não por IP fixo.

## Passo 6: Persistência com Volumes

### Atualizando o docker-compose.yml

Vamos adicionar um volume nomeado para demonstrar persistência:

```bash
code docker-compose.yml
```

**Adicione ao serviço app:**

```yaml
version: '3.8'

services:
  web:
    image: nginx:alpine
    container_name: web-frontend
    ports:
      - "8080:80"
    volumes:
      - ./app:/usr/share/nginx/html
    networks:
      - app-network
    restart: always
    depends_on:
      - app

  app:
    image: httpd:alpine
    container_name: app-backend
    ports:
      - "8081:80"
    volumes:
      - app-data:/usr/local/apache2/htdocs
      - ./app:/usr/local/apache2/htdocs
    networks:
      - app-network
    restart: always
    environment:
      - ENV=development
      - SERVICE_NAME=backend-api

networks:
  app-network:
    driver: bridge

volumes:
  app-data:
    driver: local
```

### Aplicando as Mudanças

```bash
# Parar o stack atual
docker compose down

# Iniciar com as novas configurações
docker compose up -d
```

### Testando a Persistência

```bash
# Criar arquivo no volume
docker exec -it app-backend sh -c "echo 'Dados persistentes\!' > /usr/local/apache2/htdocs/teste.txt"

# Verificar se o arquivo existe
curl http://localhost:8081/teste.txt

# Recriar o container
docker compose down
docker compose up -d

# Verificar se o arquivo ainda existe
curl http://localhost:8081/teste.txt
```

### Tipos de Volume no Compose

```yaml
volumes:
  # Bind mount - mapeia diretório do host
  - ./app:/usr/share/nginx/html
  
  # Volume nomeado - gerenciado pelo Docker
  - app-data:/data
  
  # Volume anônimo - temporário
  - /tmp
  
  # Volume com opções específicas
  - type: bind
    source: ./app
    target: /usr/share/nginx/html
    read_only: true
```

### Profiles para Diferentes Ambientes

```yaml
services:
  web:
    # ... configuração básica
  
  debug:
    image: nginx:alpine
    profiles: ["debug"]
    ports:
      - "9090:80"
```

```bash
# Rodar apenas serviços básicos
docker compose up -d

# Rodar com profile debug
docker compose --profile debug up -d
```

### Health Checks

```yaml
services:
  web:
    image: nginx:alpine
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 40s
```

### Scaling Horizontal

```bash
# Escalar serviço para 3 instâncias
docker compose up -d --scale app=3
```

## 🧹 Seção de Limpeza

### Como Desfazer Tudo que Foi Feito

Se você quiser remover completamente tudo que foi criado neste lab, siga os passos abaixo:

#### 1. Parando e Removendo o Stack Compose

```bash
# Navegar para o diretório do projeto
cd ~/ImplantacaoSistemas/Aula03

# Parar e remover todos os containers, redes e volumes
docker compose down -v
```

**O que faz:**
- `down`: Para e remove containers e redes
- `-v`: Remove também os volumes nomeados

#### 2. Removendo Imagens Baixadas

```bash
# Listar imagens usadas no lab
docker images | grep -E "nginx|httpd"

# Remover imagens específicas
docker rmi nginx:alpine
docker rmi httpd:alpine

# Ou remover todas as imagens não utilizadas
docker image prune -a -f
```

#### 3. Limpeza Completa do Docker

```bash
# Remove todos os containers parados
docker container prune -f

# Remove todas as redes não utilizadas
docker network prune -f

# Remove todos os volumes não utilizados
docker volume prune -f

# Limpeza completa do sistema
docker system prune -a -f --volumes
```

> [!CAUTION]
> **ATENÇÃO:** O comando `docker system prune -a -f --volumes` remove TUDO que não está sendo usado.

#### 4. Removendo Arquivos do Projeto

```bash
# Remover diretório completo do projeto
rm -rf ~/ImplantacaoSistemas

# Ou manter estrutura e remover apenas conteúdo
rm ~/ImplantacaoSistemas/Aula03/docker-compose.yml
rm -rf ~/ImplantacaoSistemas/Aula03/app
```

#### 5. Verificando a Limpeza

```bash
# Verificar se não há containers rodando
docker ps -a

# Verificar imagens restantes
docker images

# Verificar volumes
docker volume ls

# Verificar redes personalizadas
docker network ls | grep -v "bridge\|host\|none"

# Verificar uso de espaço do Docker
docker system df
```

#### 6. Desinstalando Docker Compose (Opcional)

```bash
# Remover plugin do Docker Compose
sudo apt remove docker-compose-plugin -y

# Verificar remoção
docker compose version
# Deve retornar: command not found
```

### Script de Limpeza Automática

Crie um script para automatizar a limpeza:

```bash
# Criar script de limpeza
cat << 'EOF' > ~/cleanup-lab03.sh
#!/bin/bash
echo "=== LIMPEZA LAB 03 ==="

# Navegar para diretório do projeto
cd ~/ImplantacaoSistemas/Aula03 2>/dev/null

# Parar stack se existir
if [ -f "docker-compose.yml" ]; then
    echo "Parando stack Docker Compose..."
    docker compose down -v 2>/dev/null
fi

# Remover imagens do lab
echo "Removendo imagens..."
docker rmi nginx:alpine httpd:alpine 2>/dev/null

# Limpeza geral
echo "Limpeza geral do Docker..."
docker system prune -a -f --volumes

# Remover arquivos do projeto
echo "Removendo arquivos do projeto..."
rm -rf ~/ImplantacaoSistemas/Aula03

echo "Limpeza concluída!"
echo "Espaço liberado:"

EOF

# Tornar executável
chmod +x ~/cleanup-lab03.sh

# Executar
~/cleanup-lab03.sh
```

### Verificação Final da Limpeza

Para confirmar que tudo foi removido:

```bash
# Verificar se não há containers relacionados ao lab

# Verificar se não há volumes do projeto

df -h
```

### Por que Fazer a Limpeza?

- **Economia de espaço**: Docker pode consumir muito espaço em disco
- **Organização**: Mantém o ambiente limpo para próximos labs
- **Performance**: Menos recursos consumidos pelo Docker
- **Segurança**: Remove containers e volumes que podem conter dados sensíveis
- **Aprendizado**: Entender como reverter configurações é importante

### Comandos de Limpeza Rápida

```bash
# Limpeza rápida (mantém imagens base)

# Limpeza completa (remove tudo)
```
